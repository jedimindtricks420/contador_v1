# Техническое описание системы

## Технологический стек
- **Framework:** Next.js 15+ (App Router, Turbopack)
- **Database:** PostgreSQL 16
- **ORM:** Prisma 6.2.1
- **Styling:** Tailwind CSS 4
- **Precision:** `Decimal.js` (все финансовые вычисления)
- **State Management:** TanStack Query (React Query) v5

## Архитектура
Система построена на принципах **двойной записи (Double-entry bookkeeping)**.
Каждая транзакция атомарно изменяет два счета: дебет и кредит.

### Схема базы данных (основные модели)
- `Account`: План счетов (Код, Наименование, Тип: Активный/Пассивный).
- `Transaction`: Проводки. Хранят ссылки на счета, сумму (Decimal), период и статус удаления.
- `Counterparty`: Справочник контрагентов.
- `SystemSettings`: Настройки организации (ID организации, `closed_period_date`, `opening_balance_date`, `is_initial_balance_fixed`).

### Логика расчетов (Core Logic)
Централизованная логика находится в `src/lib/accounting-logic.ts`:
- **Валидация (validateTransaction)**: Проверка даты на закрытый период, проверка использования счета 0000 только на дату открытия, запрет проводок 0000/0000.
- **ОСВ (Trial Balance)**:
    - **S1 (Начальное сальдо)**: Сумма всех проводок (Дт - Кт) за дату `opening_balance_date` по данному счету.
    - **Обороты**: Сумма всех проводок за выбранный диапазон дат, **исключая** дату `opening_balance_date`.
    - **S2 (Конечное сальдо)**: S1 + Обороты.
- **Изоляция счета 0000**: Все отчеты (Balance, P&L, OSV) принудительно фильтруют операции по счету 0000, чтобы технические записи не попадали в финансовые результаты.

## Безопасность
- **Middleware:** Все маршруты защищены `src/middleware.ts`, который проверяет Basic Auth заголовки.
- **Validation:** Входные данные проверяются схемой `Zod` перед попаданием в базу.
- **Soft Delete:** Поле `is_deleted` исключает случайную потерю данных.
