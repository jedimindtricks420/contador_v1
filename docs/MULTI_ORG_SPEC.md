# Техническое задание (ТЗ): Поддержка нескольких организаций и системы входа

## 1. Общие сведения
**Цель:** Перевести систему с учета одной компании на работу с несколькими независимыми юридическими лицами (Multi-tenancy) в рамках одного пользовательского профиля. Заменить базовую HTTP-авторизацию (Basic Auth) на полноценную страницу входа.

## 2. Архитектурные изменения

### 2.1. Схема базы данных (Prisma)
Необходимо внедрить модель `Organization` и связать все существующие данные с конкретной организацией.

- **Новая модель `Organization`:**
  - `id`: String (UUID)
  - `name`: String (Наименование организации)
  - `inn`: String (ИНН организации)
  - `createdAt`: DateTime
- **Связи (ForeignKey):**
  - Добавить `organization_id` в модели: `Transaction`, `Counterparty`, `Account`, `SystemSettings`.
  - Удалить глобальные настройки и сделать их уникальными для каждого `organization_id`.

### 2.2. Уровень доступа (Multi-tenancy)
- Каждое API-запрос должен автоматически фильтровать данные по `active_organization_id`.
- Идентификатор активной организации должен храниться в защищенной Cookie.

## 3. Функциональные требования

### 3.1. Аутентификация (Auth)
- **Страница входа:** Создать `/login` с монохромным дизайном.
- **Механизм доступа:** Использовать сессионный подход (JWT или Next-Auth).
- **Middleware:** При попытке входа на любую страницу без активной сессии — редирект на `/login`.

### 3.2. Переключатель организаций
- В боковом меню (Sidebar) должен появиться выпадающий список (Organization Switcher).
- При смене организации:
  1. Обновляется Cookie `active_organization_id`.
  2. Происходит перезагрузка текущей страницы для сброса кеша и загрузки новых данных.

### 3.3. Управление организациями
- Создать страницу **/organizations**:
  - Список всех юрлиц.
  - Возможность добавления новой организации.
  - Форма редактирования реквизитов (Название, ИНН).

## 4. UI / UX Требования
- **Стилистика:** Сохранить строгий monochrome (Notion-style).
- **Компонент Switcher:** Выполнить в виде компактного элемента над кнопкой «Выйти». Должен отображать текущее выбранное название компании.
- **Разделение данных:** При переключении организации Баланс, ОСВ и Журнал должны мгновенно отображать только данные выбранного юрлица.

## 5. План реализации (Roadmap)

1. **Этап 1: Миграция БД.** Создание таблиц и обновление существующих записей (назначение им дефолтной организации №1).
2. **Этап 2: Auth Migration.** Внедрение системы входа и Middleware.
3. **Этап 3: Tenant Isolation.** Переписывание API маршрутов: `where: { organization_id: activeId }`.
4. **Этап 4: Интеграция UI.** Добавление переключателя в Sidebar и страницы управления компаниями.
