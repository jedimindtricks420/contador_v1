## 4. Ядерная бухгалтерская логика (Алгоритмы)

### 4.1. Алгоритм ОСВ (Trial Balance)
Методика расчета сальдо для обеспечения точности до копейки:
1.  **S1 (Начальное сальдо)**: Сумма всех транзакций ДО `startDate` периода.
    *   `s1_net = Sum(Debit) - Sum(Credit)`
2.  **Типизация баланса**:
    *   **ACTIVE**: Если `s1_net > 0`, то `Dr = s1_net`, `Cr = 0`. Иначе `Dr = 0`, `Cr = 0`.
    *   **PASSIVE**: Если `s1_net < 0`, то `Cr = abs(s1_net)`, `Dr = 0`. Иначе `Dr = 0`, `Cr = 0`.
    *   **ACTIVE_PASSIVE**: Развернутое сальдо (Дт для положительного, Кт для отрицательного).
3.  **Обороты**: `Turn_Dr = Sum(Debit)` и `Turn_Cr = Sum(Credit)` строго ВКЛЮЧАЯ границы периода.
4.  **S2 (Конечное сальдо)**: 
    *   `Total_Dr = s1_dr + Turn_Dr`
    *   `Total_Cr = s1_cr + Turn_Cr`
    *   Применяется та же логика типизации, что и для S1.

### 4.2. Обработка счета 0000
*   Счет `0000` не включается в ОСВ и Баланс.
*   Его единственная роль — балансировка начальных проводок.
*   При `is_initial_balance_fixed = true` в `SystemSettings`, любые операции с датой `<= opening_balance_date` блокируются на уровне API.

## 6. Механизм авторизации и контекста
1.  **Session**: При логине создается JWT (jose), содержащий `userId` и `organizationId`.
2.  **Cookie**: Токен хранится в `httpOnly` куке `session`.
3.  **Middleware**: Перехватывает каждый запрос, декодирует JWT. Если токен валиден — пропускает, иначе — редирект на `/login`.
4.  **Context**: Серверная функция `getActiveOrganizationId()` извлекает ID компании из текущей сессии для фильтрации всех Prisma-запросов.

## 7. Инициализация (Seeding)
Для развертывания «с нуля» необходимо выполнить `npx prisma db seed`. Скрипт:
1. Создает системного пользователя (admin).
2. Создает дефолтную организацию.
3. Наполняет таблицу `Account` стандартным планом счетов (из `src/lib/seed-utils.ts`).
4. Устанавливает начальные `SystemSettings`.

## 8. Пользовательские сценарии (User Flows)

### 8.1. Сценарий: Инициализация и Ввод остатков (1C-style)
Это критический этап для обеспечения целостности данных при старте системы.
1.  **Авторизация**: Вход в систему через `/login`.
2.  **Настройка**: Установка `opening_balance_date` (дата, на которую фиксируются остатки).
3.  **Ввод данных**: 
    *   Для активов: Проводки вида `Дт [АКТИВ] / Кт 0000`.
    *   Для обязательств/капитала: Проводки вида `Дт 0000 / Кт [ПАССИВ]`.
4.  **Валидация**: Мониторинг виджета в Журнале операций. При достижении `Разница = 0.00` считается, что баланс сошелся.
5.  **Фиксация**: Нажатие кнопки «Зафиксировать остатки». Это переключает `is_initial_balance_fixed` в `true` и блокирует изменение проводок на дату открытия и ранее.

### 8.2. Сценарий: Операционная деятельность
1.  **Журнал**: Создание новой операции.
2.  **Интеллектуальный выбор**: Использование `SearchableSelect` для поиска счетов по коду. Визуальные индикаторы (цвета) помогают избежать ошибок выбора типа счета.
3.  **Контрагенты**: Привязка операции к конкретному партнеру для аналитики задолженности.
4.  **Период**: Система автоматически подставляет `period` (ММ.ГГГГ) на основе даты операции.

## 9. Математическое ядро (Mathematical Core)

### 9.1. Балансировка вспомогательного счета 0000
Формула для проверки корректности ввода остатков:
$$ Difference = \sum_{i=1}^{n} Amount_{Dr=0000} - \sum_{j=1}^{m} Amount_{Cr=0000} $$
*Важно: Система разрешает фиксацию только при Difference = 0.*

### 9.2. Развернутый алгоритм ОСВ (Trial Balance)
Для каждого счета $A$ в интервале дат $[D_{start}, D_{end}]$:
1.  **Начальное сальдо (S1)**: 
    *   $Dr_{base} = \sum Amount(Dr=A, Date < D_{start})$
    *   $Cr_{base} = \sum Amount(Cr=A, Date < D_{start})$
    *   Для `ACTIVE`: $S1\_Dr = \max(0, Dr_{base} - Cr_{base}), S1\_Cr = 0$.
    *   Для `PASSIVE`: $S1\_Cr = \max(0, Cr_{base} - Dr_{base}), S1\_Dr = 0$.
2.  **Обороты (Turnover)**:
    *   $Turn\_Dr = \sum Amount(Dr=A, D_{start} \le Date \le D_{end})$
    *   $Turn\_Cr = \sum Amount(Cr=A, D_{start} \le Date \le D_{end})$
3.  **Конечное сальдо (S2)**:
    *   $Net = (S1\_Dr + Turn\_Dr) - (S1\_Cr + Turn\_Cr)$
    *   Применяется логика знака сальдо аналогично S1.

### 9.3. Агрегация Баланса (Form 1)
Баланс формируется путем группировки сальдо по префиксам кодов:
*   **Активы (Assets)**: 
    *   Денежные средства (50xx, 51xx)
    *   Дебиторская задолженность (40xx)
    *   ТМЗ (29xx)
*   **Пассивы (Liabilities & Equity)**:
    *   Кредиторская задолженность (60xx)
    *   Собственный капитал (83xx)
    *   Нераспределенная прибыль (87xx)

### 9.4. Финансовый результат (P&L / Form 2)
Расчет выполняется на основе оборотов транзитных счетов:
$$ NetProfit = Turnover_{Cr}(9010) - \sum Turnover_{Dr}(94xx) $$
Где 9010 — Выручка, а 94xx — Операционные расходы (Маркетинг, Администрирование и т.д.).
